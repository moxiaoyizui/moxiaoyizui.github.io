一个底层小码农对「业务」的认知更迭

Posted on 2021-06-01 Edited on 2021-06-02 
 众所周知, 物理学的研究分为理论物理与应用物理两个流派, 并且常年打的不可开交, 
 两边都觉得对方是异端, 理论物理认为应用太俗; 应用物理认为理论太装, 恨不得人脑子打出狗脑子来
 事实上, 写代码的, 或者说搞技术的, 也有类似的两个方向, 一般称为学术型和工程型.
 本文接下来所讨论的, 都是建立在工程型的基础上.

程序员的自我修养

程序员最基本的素质是什么? 毋庸置疑, 只有一个答案, 那就是秃头

程序员最核心的竞争力是什么?

技术.

吗?

曾经的我就是这么认为的. 这个曾经指的大概是我工作的前三年

曾经的我坚定不移的认为, 一个NB的程序员只要技术到位就可以了, 别管什么Java, C艹, python, go, 全都搞定, 出什么新的技术就去学什么, 学到会为止, 直到万剑归一, 天下大同, 即可成就无上金身

but, 事实真的如此吗?

卑微小杨的认知进化

阶段-1 无知的蚂蚁

第一次感觉不对, 发生在工作两年多的时候. 当时我发现, 我已经掌握的技术, 已经可以搞定日常工作百分之九十的需求.

于是我得出结论: 需求太简单了, 完全发挥不了我吊炸天的技术. (现在回过头去想, 只能用一个字形容当时的我了: **)

接下来呢? 接着奏乐, 接着舞. 我行我素该咋咋地.

阶段0 水缸里的小跳蛙

第二次发现问题, 是在我开始做营销系统之后

相对于最开始做的项目来说, 营销系统是一个更整体, 业务上更复杂的服务

但是做起来的感觉是什么呢? “明明身负不世之力, 奈何身陷桎梏无处屠龙”

翻译成人话就是, 明明感觉技术够用, 但做出来的东西却根本不能算个玩意儿

终于有一次在被阴阳人运营亲切关心后, 默默的思考了一个下午

我是不是漏掉了什么?

对于业务会基本是得过且过, 完全不过脑子;

接了需求之后直接照猫画虎. 抽象? 拓展? 不存在的, 实现就好

这样写出来的代码是什么样的呢?

简洁, 但单一

明确, 但不可复用

只要需求有一点改动, 基本就要全都再来一遍

扪心自问, 这东西确实连屎山都算不上, 毕竟屎还可以用来施肥

在抽了根烟之后, 我明白我遗漏的是什么了

– 业务.

阶段1 名叫Alex的非洲灰鹦鹉

这是我第一次, 把懂业务也列入程序员的技能范畴里, 但也只是单纯的列进来了而已, 相比技术本身, 相差了大概十个优先级

具体体现在我接了需求之后, 会去思考一下, 它到底是要做什么, 是不是可以有更好的方案. 如果之后需求有迭代, 我是否可以在本次设计上cover住

立竿见影, 最直观的体现就是我写的代码可以复用了! 恨不得放几挂鞭炮庆祝一下

但是问题并没有完全解决, 甚至随着我要做的东西越来越多, 反而更加严重了

之前是体现在代码不能复用, 现在是体现在功能,模块不能复用

但这个时候我已经稳如老狗, 丝毫不慌

毕竟我已经看见了这个世界

阶段2 人类

顺着之前的思路, 开始逐渐深入业务

先是纵向, 继而横向

于是腰也不酸了, 腿也不痛了, 做起技术方案和设计也更加有劲了

甚至我认为, 在某些时候, 一个程序员会比PM更能做好一个产品

因为程序员可以真正的了解底层, 从而自下而上, 形成一个完整的闭环

我见过很多程序员转了PM, 每一个都能做到标准线以上

但是我没见过PM成功转开发的案例

抛开这些疑似诋毁小伙伴的观点, 我得出了结论

做一个工程型的程序员, 必须要懂业务, 而且要把业务跟技术放在同样重要的地位

不懂业务, 不能称为一个真正的程序员

阶段3 半仙

这是我目前还没有达到的一个阶段, 也是靠接触其他大佬, 才有一些模糊的方向

具体一点, 我认为这个阶段的人就是架构师, 业务专家等角色

在这个阶段, 不是简单的结合业务做设计

用在QCon上听滴滴某大佬的观点, 要达到这个阶段, 需要形成“支撑业务 -> 提升业务经营效率 -> 创造业务新机遇 ->”这样一条闭环

在这一阶段, 甚至业务要比技术还要重要那么一捏捏

final 陆地神仙

这个阶段, 我认为是终极理想, 但是永远不会有人达成.

基于上述观点, 达到这一阶段的人需要完全知悉业务现状, 发展路线, 未来趋势

此上种种, 非人力所能及也

技术和业务的究竟是个怎样的关系

先说技术.

正如我上面说的, 对于工程型程序员来说, 技术是有极限的

这个极限指的不是人的极限, 而是落地的极限

人会的技术再多再牛逼, 具体需求是select * from table where id = 1, 也是屠龙之技

而业务是什么?

在之前我并没有明确的定义

一直以来, 我认为业务是可以让一个coder更好的做设计, 一种让技术落地的手段

这个观点对么? 不能说错

但是在一场技术大会上听了几场关于架构师还有业务的演讲之后, 开始感觉有哪里不对

当天晚上回家, 吃完饭打一把dota, 突然发现问题在哪儿

如果类比到Dota, 技术就是选手的个人实力, 和英雄本身的模型

业务就是战术, 运营

Dota作为一个十多年的老游戏, 发展到如今, 一流战队之间的能力差距已经不是那么大

能力差的队伍根本没有进入一流的机会

而能在其中拔得头筹的, 无一不是战术领先, 运营得当的队伍

为什么? 因为这样可以在一场比赛里获得更多的资源

技术再nb, 被对手领先三把圣剑两个龙心, 终究还是被碾成渣渣

所以, 业务与其说是让技术落地的手段, 不如说是业务让技术真正的产生了价值

技术决定下限, 业务决定下限
